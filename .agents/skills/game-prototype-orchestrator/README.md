# 🎮 Game Prototype Orchestrator

**Game Prototype Orchestrator** (`game-prototype-orchestrator`) serves as the master routing module within the Game Pipeline ecosystem. Its primary responsibility is to automatically convert comprehensive game documentation (such as Game Design Documents - GDDs - and Asset configuration files) into fully functional, playable Unity prototypes. It achieves this by orchestrating a precise sequence of specialized sub-skills and leveraging the **MCP (Model Context Protocol)** to directly interface with and control the Unity Engine.

The defining characteristic of this skill is its **"Fire-and-Forget" execution model**. Once triggered, the AI assumes total autonomy—planning logic, dividing the workload into highly granular task files, and pushing direct C# code and Editor commands to Unity—without pausing for continuous human intervention between execution checkpoints.

---

## ⚙️ The 3-Phase Execution Flow (Sub-skills Sequence)

This orchestrator operates strictly sequentially. Under no circumstances does it skip steps or execute tasks simultaneously. It relies on the rigorous execution of three distinct sub-skills in the exact order below:

### Phase 1: Ingestion & Verification (`subskills/01_Ingestion_Verification.md`)
- **Objective:** Establish the foundation, read requirements, and verify the technical context.
- **Process:** The AI reads the provided `GDD.md` and the associated project asset lists (e.g., `.json` files). It scans and maps all predefined mechanics, entities, and visual/audio assets.
- **Placeholder Management:** If any visual or audio assets are currently missing (not yet generated by the Art/Audio pipelines), the system automatically maps and generates **Placeholders** (such as dummy prefabs or solid-colored primitives) to ensure the engineering pipeline is not bottlenecked.
- **Gatekeeper Rule:** The orchestrator *must wait* to ensure all assets are fully accounted for, and all necessary placeholders are mapped, before it is allowed to proceed to Phase 2.

### Phase 2: Ahead-of-Time Task Generation (`subskills/02_Task_Generation.md`)
- **Objective:** Prevent context loss ("hallucination" or "forgetfulness") during long execution chains by pre-computing all development steps.
- **Process (CRITICAL):** The AI dissects the monolithic GDD into ultra-granular, component-level tasks. These tasks are saved as physical, individual Markdown files inside the project's task directory: `Assets/Projects/[ProjectName]/.agents/tasks/` (e.g., `Pack_A_Setup.md`, `Pack_B_GameplayCore.md`).
- **Gatekeeper Rule:** The orchestrator *must wait* until the entire task list is physically written and confirmed present in the `.agents/tasks/` folder. It cannot begin coding while it is still writing the blueprint.

### Phase 3: Automated MCP Execution & Reporting (`subskills/03_MCP_Execution.md`)
- **Objective:** Physical manifestation of the game prototype via Unity Editor control.
- **Process:** The AI iteratively reads each `.md` task file generated during Phase 2 and translates them into actionable engineering commands transmitted directly to Unity via MCP. This includes C# script generation, hierarchy manipulation, component attachment, applying Unity physics rules, and managing object references via absolute hierarchy paths.
- **Execution Rule:** This step strictly enforces the **"Fire-and-Forget"** philosophy. The AI *must not* halt to ask for human review between handling individual task files. It executes the entire backlog autonomously, utilizing automated `try-catch` structures to prevent cascading compilation errors resulting from domain reloads.
- **Completion:** Once every task file has been processed and executed, the orchestrator formally halts and reports back to the user that the prototype is ready for manual human inspection.

## 🚀 How to Use (Operator Guide)

### Prerequisites
1. **Valid Project Data:** The target project must contain a well-structured `GDD.md` and an asset registry located within its workspace (e.g., `Assets/Projects/[ProjectName]/`).
2. **Environment Setup:** The Unity Editor must be active with the target project open, and the Unity MCP Server must be actively running in the background to receive commands.

### Triggering the Skill
You can invoke this orchestrator directly from the agent interface using natural language alongside the explicit skill tag.

**Command syntax:**
> `@/game-prototype-orchestrator Read the GDD in the FlappyTrippy project and initiate the prototype build.`
> 
> `@/game-prototype-orchestrator Please start the prototype phase for the current active project.`
